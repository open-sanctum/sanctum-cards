- Regenerated `data/cards.json`, `data/cards/<id>.json`, `data/enums.json` with the new fields populated

## 4. Recommended approach: pre-analyze outside the agent, then dispatch a small focused agent

### Phase 1 — Pre-analysis (no LLM tokens)

Done by the human or a one-shot bash command BEFORE any agent is dispatched. This is just Ghidra doing CPU work; no model attention needed.

```bash
# Extract the debug binary if not already extracted
mkdir -p /tmp/re && cd /tmp/re
unzip -o /home/dmm/src/sanctum-cards/inputs/Sanctum18-04.zip \
  'Sanctum18/bin/Sanctum-Debug-2008.exe' -d .

# Run Ghidra headless analysis (10-20 minutes)
mkdir -p /tmp/re/ghidra-project
analyzeHeadless /tmp/re/ghidra-project sanctum \
  -import /tmp/re/Sanctum18/bin/Sanctum-Debug-2008.exe \
  -analysisTimeoutPerFile 1800 \
  -log /tmp/re/ghidra-analysis.log

# Verify it succeeded
tail /tmp/re/ghidra-analysis.log
# Should end with "Saving project: /tmp/re/ghidra-project/sanctum" or similar
```

The output is a complete `.gpr` project at `/tmp/re/ghidra-project/sanctum.gpr` with:
- All functions identified
- RTTI parsed (class hierarchies, vtables)
- Cross-references built
- Strings collected

This pre-analysis is **the most important step**. Without it, every agent dispatch starts from scratch and burns context waiting for analysis to finish.

### Phase 2 — Targeted extraction (small focused agent, 5-15 minutes)

With the analyzed project ready, dispatch a focused agent. The agent's task is narrow:

1. Locate the `CSpellCost::CSpellCost` constructor symbol via the symbol table.
2. Find cross-references to the constructor — these are the static-initializer call sites (one per card).
3. For each xref:
   - Read the `this` pointer at the call site (the static struct's address)
   - Read the constructor arguments (the cost fields)
   - Decode the struct layout (likely 6 int fields = one count per mana index, OR 3 pairs of (mana_index, amount), OR similar)
4. Cross-reference with known cards (e.g. Wrack id 4, Nightmare id 1026) to determine the mana-index → mana-name mapping.
5. Emit `inputs/cards-from-binary.json` with one record per card id.
6. Write ADR-0002 documenting the binary layout discovered.

Skeleton Ghidra script (Jython 2.7 — Ghidra's scripting language is NOT Python 3):

```python
# tools/re/extract_cost_table.py
# Run via:
#   analyzeHeadless /tmp/re/ghidra-project sanctum \
#     -process Sanctum-Debug-2008.exe \
#     -postScript /tmp/re/extract_cost_table.py
import json

prog = currentProgram
st = prog.getSymbolTable()
ref_mgr = prog.getReferenceManager()
listing = prog.getListing()
memory = prog.getMemory()

# 1. Find CSpellCost class namespace
cspell_class = None
for ns in st.getClassNamespaces():
    if ns.getName() == "CSpellCost":
        cspell_class = ns
        break
assert cspell_class is not None, "CSpellCost class not found — was the project analyzed?"

# 2. Find the constructor (mangled vs. demangled; try both)
ctor = None
for sym in st.getSymbols(cspell_class):
    name = sym.getName()
    if name == "CSpellCost" or "ctor" in name.lower():
        ctor = sym
        break
assert ctor is not None, "CSpellCost::CSpellCost constructor not found"

# 3. Walk xrefs (callers)
records = []
for ref in ref_mgr.getReferencesTo(ctor.getAddress()):
    call_addr = ref.getFromAddress()
    # The `this` pointer and constructor args are PUSHed onto the stack
    # immediately before the CALL. Walk back ~10 instructions and
    # decode the PUSHes.
    instr = listing.getInstructionAt(call_addr)
    args = []
    for _ in range(10):
        instr = instr.getPrevious()
        if instr is None: break
        mnem = instr.getMnemonicString()
        if mnem == "PUSH":
            op = instr.getOpObjects(0)
            if op and len(op) == 1:
                args.append(op[0])
        elif mnem == "CALL":
            break  # hit a previous call; stop walking
    # First arg pushed (last in code order) is `this`; rest are cost fields.
    # Adjust based on what we observe.
    records.append({
        "call_addr": str(call_addr),
        "args": [str(a) for a in args],
    })

# 4. Output for analysis
print(json.dumps(records[:5], indent=2))  # peek
with open("/tmp/re/cspellcost-raw.json", "w") as f:
    json.dump(records, f)
print("Wrote %d records to /tmp/re/cspellcost-raw.json" % len(records))
```

**Critical**: this is a STARTING POINT. Real implementation will need refinements:
- Constructor name might be mangled differently (e.g. `??0CSpellCost@@QAE@...`)
- Args may be passed via registers (thiscall on i386 uses ECX for `this`)
- Walking PUSHes is fragile; Ghidra's decompiler view is more robust
- Use the decompiler API (`DecompInterface`) instead of raw instruction walking for cleaner arg extraction

### Phase 3 — Verification (inline, no agent)

Cross-check the extracted JSON against known cards by hand:

```bash
# Card 4 (Wrack) — alteration, minion target, simple Wrack-like card

# Card 1026 (Nightmare) — summoning, HP 10 L 2 monster; should cost more
jq '.[] | select(.id == 1026)' inputs/cards-from-binary.json

# Spot-check distribution
jq '[.[].cost.total] | sort | unique' inputs/cards-from-binary.json
# Should be a sensible range (0 to ~MaxSymCostCard)
```

Determining the mana-index → name mapping:

- Take 2-3 cards whose rules text reveals their primary mana (e.g. "Caster gets +1 Order" hints at Order in the cost; "+2 Will and -1 Clarity" hints at Will/Clarity).
- Look at their extracted cost records; the index that appears with the highest amount is the primary mana for that card.
- Cross-reference across 5+ cards to lock in the 0-5 mapping.

If the mapping is ambiguous after cross-referencing, fall back to: cards from the Life starter deck (`Decks/PRECONSTRUCTED/LIFE_Starter.dck`) should predominantly use Will + Order (Life's two mana types per game lore).

## 6. Risks / open uncertainties

1. **Mana-index ordering** — internal order may not be alphabetical. Resolved during Phase 3 verification by cross-referencing known cards.
2. **`CSpellCost` record layout** — may be 6 ints (one per mana type), or 3 (type, amount) pairs, or a packed bitfield. Ghidra's structure view should expose it directly.
3. **Indirect construction** — the constructor might be called via a factory, in which case xref walking finds the factory, not the static initializers. Mitigation: look for the `StaticCosts` global and its initializer instead.
4. **`TAlterSpellCostIf` dynamic adjustments** — some cards mutate their base cost. We extract the BASE cost; runtime adjustments are out of scope (sub-project C / rules engine).
5. **No rarity / set in `CSpellCost`** — these fields might live in a different struct (e.g. `CCardData` per the source path `carddata_cspell.cpp`). If `CSpellCost` only holds mana counts, we'd need a parallel investigation for the `CCardData` static table.
